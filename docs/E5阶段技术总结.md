# E5阶段目标
主线：
- health check 判断新系统是否健康
- 成功后 mark-good
- 失败后不 mark-good / mark-bad / reboot
- watchdog 负责 health check 卡死后的 reset 兜底

内核子线：
- 裁剪并验证 MMC、ext4、UART、watchdog 等 OTA 必需内核能力
- 不做激进裁剪
- 每裁掉一类功能，都必须证明 E1~E4 主链路仍可用

# health check
我们编写了两个关于健康检查的脚本
- /etc/init.d/S99edgeguard-health
 - 在系统启动后init.d的S99脚本自动启动，进行自动的健康检查.
 - 负责检查/data分区是否被挂载，以及一些flag文件是否存在.
 - 调用edgeguard-health-check函数，并生成对应的flag文件信号供watchdog进行判断.
- /usr/bin/edgeguard-health-check
 - 进行具体的健康检查,内容包括(系统启动次数是否异常，当前启动槽位与预期槽位是否相同，当前系统版本是否可获得，是否符合预期).
 - 根据健康检查情况，生成一个对应的flag文件供watchdog部分使用，并决定是对该槽位mark-good还是fail.

# watchdog 
etc/init.d/S98edgeguard-watchdog的工作流如下：
- 打开 /dev/watchdog
- 健康检查还没结束时，周期性喂狗
- 看到 /run/edgeguard/health.ok 后，继续喂狗
- 看到 /run/edgeguard/health.fail 后，停止喂狗，等待 watchdog 复位
- 超过 HEALTH_TIMEOUT 还没看到 ok/fail，也停止喂狗，等待 watchdog 复位

## 现行方案优点
相比只使用S99作为自动健康检查。使用看门狗作为复位扳机，除了健康检查失败能回滚之外，健康检查卡死、启动流程卡住、手动 reboot 失败 这类异常也能通过 watchdog 兜底复位。当健康检查进程因为未知原因阻塞，原先方案无法继续mark-good or mark bad。系统健康检查流程被卡死。使用看门狗做扳机。当在期望时间内健康检查进程被阻塞无法发出检查结果信号，watchdog能够实现复位兜底。

## 现行方案不足

edgeguard-watchdog-keeper 会一直运行并持续喂狗。也就是说，一旦启用 enable_keeper，它不是只在健康检查期间运行，而是健康检查成功后也持续运行。如果系统后续真正的主业务进程挂了，但 keeper 还活着并继续喂狗，watchdog 不会复位设备。这适合“只保护启动阶段”的设计，但不适合“保护主业务运行状态”的设计。

如果业务目标是保护主业务进程，那么 keeper 不应该只看 /run/edgeguard/health.ok，还应当周期性检查业务服务状态，例如主进程 PID、心跳文件、关键接口响应等。

# 内核裁剪
## 如何判断该裁剪什么模块？
### 第一步：建立“不可裁剪基线”。

先不要研究每个 CONFIG_，先写一张“系统必须活着”的清单。你的 E5 类设备至少应把这些列为硬保留：启动介质 MMC/eMMC、分区、ext4、devtmpfs、procfs、sysfs、串口、serial console、watchdog、根文件系统挂载、/data ext4、RAUC 安装 bundle 所需能力、故障回滚所需能力、日志和调试所需的最小能力。

这一步的规则是：凡是属于“启动、挂载、控制台、看门狗、OTA、回滚、救砖、量产测试”的配置，默认保留。只有明确证明它不在链路上，才考虑动。

## 第二步：从 .config 反查 Kconfig 定义。
prompt      菜单里显示什么名字
help        这个功能到底做什么
depends on  它依赖谁
select      它会强制打开谁

## 编译好了镜像该怎么烧入
- 先在主机端将zImage复制到/data下
- 将 BOOT 分区挂载到 rootfs 目录树里的挂载点/mnt。
- 把zImage复制到/mnt
- 重启。

- 在主机端直接把zImage复制到sd卡的BOOT分区。
